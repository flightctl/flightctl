# Schemathesis API Tests

This suite runs [Schemathesis](https://schemathesis.readthedocs.io/) against the APIs defined in our OpenAPI specs. It performs schema-based fuzzing, stateful testing (multi-step CRUD sequences), coverage analysis, and custom checks.
When new endpoints are added to the specs, they are automatically included on the next run.

## Reproducing failures

When a test fails, Schemathesis prints a `curl` command with the exact request that triggered the failure. Hypothesis automatically reduces the input to the simplest case that still reproduces the issue:

```text
Reproduce with: curl -X POST -H 'Content-Type: application/json' \
  -d '{"name": ""}' \
  https://localhost:3443/api/v1/fleets
```

Run the curl command to reproduce the failure and read the response body for details.

## How the test run works

- Deploy the backend: `make deploy`
- To apply API test overrides (rate limits, rendered spec poll timeout) to the running deployment:

  ```bash
  helm upgrade flightctl ./deploy/helm/flightctl/ -n flightctl-external --reuse-values --values test/api/values.api-tests.yaml
  ```

- `make test-api` builds the schemathesis container image and runs `test/api/run_tests.sh`.
- The runner discovers test suites from the directory structure (`core/v1alpha1`, `core/v1beta1`, `imagebuilder/v1alpha1`).
- For each suite, `run_suite.sh` runs inside the container:
  1. **Schemathesis CLI** (`st run`) with `--mode positive` against the suite's OpenAPI spec. Runs through three active phases: examples, fuzzing, and stateful.
  2. **Pytest version probes** (`test_version_probes.py`) that validate version negotiation behavior.
- A consolidated report is generated by `report.py`.

## Directory structure

```text
test/api/
  common/
    hooks.py          -- Shared hooks: TraceCov, int64 patching, protocol fields,
                         label sanitization, query parameter filtering
    checks.py         -- Custom response checks: version header, body apiVersion, Vary header
  core/
    hooks.py          -- Core-specific hooks: CSR generation, authprovider spec,
                         catalog items, fleet template, status conditions, approval
    v1alpha1/
      hooks.py        -- Version entry point (registers v1alpha1)
      schemathesis.toml
    v1beta1/
      hooks.py        -- Version entry point (registers v1beta1)
      schemathesis.toml
  imagebuilder/
    hooks.py          -- ImageBuilder hooks: fixture creation (repos, imagebuild),
                         repository and imageBuildRef injection
    v1alpha1/
      hooks.py        -- Version entry point (registers v1alpha1)
      schemathesis.toml
  conftest.py         -- Pytest fixtures for version probe tests
  test_version_probes.py -- Pytest tests: invalid version -> 406, missing version -> fallback
  run_tests.sh        -- Top-level runner (auth, URL discovery, suite iteration)
  run_suite.sh        -- Per-suite runner (schemathesis CLI + pytest)
  report.py           -- Generates consolidated markdown report from JUnit, TraceCov, and HAR data
  render_log.py       -- Strips ANSI codes from schemathesis output
  values.api-tests.yaml -- Helm overrides for API testing (rate limits, rendered spec poll timeout)
  Containerfile       -- Container image: schemathesis + openssl + pyte
```

## Test phases

Each suite supports four phases (configured in `schemathesis.toml`):

| Phase | Purpose |
|-------|---------|
| **Examples** | Sends requests using `example` values from the OpenAPI spec |
| **Coverage** | Generates test cases to maximize schema coverage (currently disabled -- takes a long time) |
| **Fuzzing** | Generates random schema-compliant data to find edge cases |
| **Stateful** | Builds multi-step sequences (create -> read -> update -> delete) using OpenAPI `links` |

## Hooks

Hooks customize data generation and response validation. They are organized hierarchically:

- **`common/hooks.py`** registers hooks shared by all suites (protocol fields, labels, query filtering).
- **Service hooks** (`core/hooks.py`, `imagebuilder/hooks.py`) add service-specific logic (CSR generation, fixture creation, field fixing).
- **Version entry points** (`v1alpha1/hooks.py`, `v1beta1/hooks.py`) import service hooks and set the API version.

## Custom response checks

Defined in `common/checks.py`, these run on every response:

1. **`check_version_header_echo`** -- `Flightctl-API-Version` response header matches the requested version.
2. **`check_body_api_version`** -- `apiVersion` in response body matches (`vX` or `flightctl.io/vX`).
3. **`check_vary_header`** -- `Vary` header includes `Flightctl-API-Version`.

## Version probe tests

`test_version_probes.py` validates version negotiation (run via pytest after schemathesis):

- Invalid version header -> `406 Not Acceptable`
- Missing version header -> response includes default version

## Prerequisites

- Running flightctl deployment (`make deploy`)
- `~/.config/flightctl/client.yaml` with valid auth token
- Podman
- Agent enrollment key at `bin/agent/etc/flightctl/certs/client-enrollment.key`

## Run

```bash
make test-api
```

To run specific suites only, set `SCHEMATHESIS_SUITES` (comma-separated):

```bash
make test-api SCHEMATHESIS_SUITES=imagebuilder/v1alpha1
make test-api SCHEMATHESIS_SUITES=core/v1beta1,imagebuilder/v1alpha1
```

## Configuration

Each suite has a `schemathesis.toml` controlling phases, checks, and generation limits. Key settings:

- `max-examples` -- Max test cases per operation (fuzzing/stateful phases)
- `phases.<name>.enabled` -- Enable/disable individual phases
- `checks.<name>.enabled` -- Enable/disable individual checks
- `codec = "ascii"` -- Restrict generated strings to ASCII

### Operation overrides

Use `[[operations]]` blocks to customize behavior for specific endpoints:

Disable a specific check for an endpoint:

```toml
[[operations]]
include-path = "/enrollmentrequests/{name}/approval"
[operations.checks.response_schema_conformance]
enabled = false
```

Disable an endpoint entirely:

```toml
[[operations]]
include-path = "/auth/{providername}/token"
enabled = false
```

Pin path parameters to specific values:

```toml
[[operations]]
include-path = "/fleets/{name}"
parameters = { "path.name" = "default" }
```

## Extend coverage

- **Add a version**: Create a new `<service>/<version>/` directory with `hooks.py`, `__init__.py`, and `schemathesis.toml`. Add the suite to `run_tests.sh`.
- **Add a service**: Create a new `<service>/` directory with `hooks.py` and version subdirectories. Register path-to-kind mappings in the service hooks. Update `run_tests.sh`.
- **Add hooks**: Add `map_body` hooks to fix generated data, or `filter_body` hooks to skip invalid cases. Service-specific hooks go in `<service>/hooks.py`.

## Results

Outputs are written to `reports/schemathesis/<service>/<version>/`:

- `st_output.log` -- Clean schemathesis output
- `junit-*.xml` -- JUnit test results
- `junit-probes.xml` -- Version probe test results
- `schema-coverage.json` / `schema-coverage.html` -- TraceCov coverage data
- `har-*.json` -- HTTP Archive (request/response capture)
- `*.json.gz` -- VCR cassettes (request/response recordings)
- `ndjson/` -- NDJSON detailed test output
- `report.md` -- Consolidated markdown report (at `reports/schemathesis/`)
